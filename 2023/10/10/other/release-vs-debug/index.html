前几天在做一个实验的时候用到了 VS2022 的调试器，突然发现用 release 版本下依然可以调试
但根据已有的浅显的知识，release 版本是优化后的版本，不会附加任何调试信息，那 VS 又是如何调试的呢？

百度之后发现说有个叫做 .pdb 的文件，在调试时会装载到文件中，以提供额外的辅助信息

光说肯定是不行的，为此我需要做几件事来解决我的疑惑：

  • 对比查看 Debug 和 Release 版本下生成的文件
  • 查看 .pdb 文件里究竟是什么
  • .pdb 是如何被装在使用的

x86_amd64: 编译器为 x86 版本，输出文件为 amd64

关于 x64 和 amd64

在 64 位指令系统中，有两家公司在做
一家是 Intel 推出的IA-64指令集，但不兼容 IA-32（即x86架构使用的）指令集
另一家是 AMD 公司推出的 AMD64 指令集，兼容了 IA-32
显而易见，IA-64会被淘汰，更何况 IA-64 比 AMD64 出现更晚

最后Intel也不得不采用 AMD64 指令系统，但换了个名字，叫x86_64架构（屏蔽掉了 AMD 的牌子），简称 x64
因此，现在来讲，x86_64、x64、AMD64基本上是一个东西

不纠结了，选择 amd64 版本的编译器

使用 Debug 模式下编译生成两个文件：

在刚才的尝试里发现了在 VSCode 里直接使用其默认配置的 Release 生成方式是不会生成 .pdb 文件的，并不能调试

具体配置参见 Set debug and release configurations in Visual Studio

关于更多编译选项如下：
Debug

  • /MDd /MLd or/MTd: Use run-time library for Debug
  • /Od: Disable optimization
  • /D “_DEBUG”: Equals to #define _DEBUG, open code like assert
  • /ZI: Open Edit and Continue database, allows us to modify code while debugging
  • /GZ: Enable stack frame Run-Time error checking. deprecated. use /RTC instead.
    Release
  • /MD /ML or/MT: Use run-time library for Release
  • /O1: Minimize size
  • /O2: Maximize speed
  • /D “NDEBUG”: Close conditional compilation (don’t compile assert)
  • /GF: Eliminate duplicate strings and pools strings as read-only. If you try to modify strings under /GF, an application error occurs.
